# Сессия: REPEAT-LOOP-01 Mastery Loop Task Pack

Дата: 2026-03-23

## Контекст
- FamilyRecipes уже хорошо решает передачу рецепта, хранение авторства и семейного контекста.
- В приложении уже есть:
  - единая карточка семейного рецепта;
  - detail screen;
  - личная заметка пользователя;
  - CTA `Попросить уточнить`.
- Следующий продуктовый шаг: помочь пользователю не просто сохранить рецепт, а довести его до состояния `у меня теперь получается`.

## Цель
Добавить в FamilyRecipes минимальный mastery loop: после готовки пользователь может отметить попытку, коротко зафиксировать результат, при необходимости задать один узкий вопрос автору и видеть, как рецепт двигается от состояния `получил` к состоянию `замастерил`.

Итог должен быть таким:
- рецепт ощущается не как статичная карточка, а как путь к повторяемому результату;
- пользователь может быстро отметить первую и повторные попытки;
- detail screen показывает простой прогресс освоения;
- неудачная или неполная попытка естественно ведёт к мягкому уточнению у автора;
- текущая модель `оригинал рецепта + личная заметка` не ломается.

## Product thesis
- Рецепт в FamilyRecipes — это не только память и не только текст.
- Рецепт — это репетиция, где через повторения, заметки и уточнения появляется устойчивый навык.
- Главная эмоция этого пакета: `не просто храню мамин рецепт, а реально научился его готовить`.

## Главная продуктовая модель

### Два разных статуса
Важно не смешивать два разных слоя:

1. **Lifecycle status карточки**
- `pending`
- `received`
- `clarification`

Это статус сущности `request/recipe card`.

2. **Mastery status рецепта**
- `получил`
- `пробовал`
- `почти получилось`
- `замастерил`

Это статус пользовательского освоения конкретного рецепта.

`mastery_status` не должен подменять текущий `card_status` и не должен ломать unified card model.

### Два разных типа заметок
Важно не смешивать:

1. **Личная заметка**
- уже существует;
- свободная, постоянная, может быть о чём угодно;
- не обязательно привязана к конкретной попытке готовки.

2. **Post-cook note**
- появляется после конкретной готовки;
- фиксирует результат попытки;
- в MVP показывается как последняя заметка после готовки.

`post-cook note` не заменяет текущую personal note и не должен храниться в том же примитиве без явного разграничения смысла.

## MVP Scope

### `REPEAT-01` Attempt logging
- На карточке полученного рецепта пользователь может нажать `Готовил(а)` или эквивалентный лёгкий CTA.
- После этого он выбирает короткий результат попытки:
  - `Не получилось`
  - `Почти получилось`
  - `Получается стабильно`
- Это должно работать и для первой, и для повторных попыток.

Публичный user-facing `mastery_status` при этом остаётся в 4 состояниях:
- `получил`
- `пробовал`
- `почти получилось`
- `замастерил`

Рекомендуемое отображение:
- `Не получилось` -> `пробовал`
- `Почти получилось` -> `почти получилось`
- `Получается стабильно` -> `замастерил`

### `REPEAT-02` Post-cook note
- После отметки попытки пользователь может сохранить короткую заметку.
- Не делать тяжёлый structured editor.
- Достаточно одного лёгкого текстового поля с подсказкой вроде:
  - что получилось;
  - что не получилось;
  - что изменить в следующий раз.

В MVP нужно сохранить:
- последнюю заметку после готовки;
- связь заметки с попыткой;
- возможность быстро посмотреть её на detail screen.

### `REPEAT-03` Mastery progress on detail
- На detail screen нужно показать:
  - сколько раз пользователь готовил этот рецепт;
  - текущий `mastery_status`;
  - последнюю post-cook note.
- Прогресс должен быть виден на экране рецепта без ощущения "журнала тренировок".
- Если list-level badge добавляется слишком дорого или шумно, detail screen важнее списка.

### `REPEAT-04` Contextual clarification
- После первой неудачной или неполной попытки показывать мягкий CTA `Уточнить у автора`.
- Не делать новый общий flow с нуля.
- Нужно переиспользовать существующий clarification request flow и `parent_recipe_id`.
- Отличие только в framing:
  - пользователь уточняет один узкий вопрос;
  - UI и prefill должны подталкивать к конкретике (`когда солить`, `какая консистенция`, `сколько жарить`), а не к размытому "пришлите рецепт заново".

## Рекомендуемый backend shape
- Не обязательно следовать буквально, если Claude найдёт чище, но продуктовый контракт должен сохраниться.

Предпочтительный путь для MVP:
- отдельная append-only сущность попыток готовки, например `recipe_attempts`;
- без полноценного journal UI;
- без versioning;
- без merge личной версии и оригинала.

Минимально нужно уметь получить для recipe detail/read-model:
- `cook_count`
- `mastery_status`
- `latest_attempt_result`
- `latest_attempt_note`
- `latest_cooked_at`

Рекомендуемая форма одной попытки:
- `recipe_id`
- `created_by`
- `result`
- `note_text`
- `created_at`

Почему не local-only:
- mastery loop — часть core product loop, а не просто supporting preference;
- нужны измеримые метрики по попыткам и переходам между состояниями;
- нужен путь к cross-device consistency;
- локальное хранение быстро превратится в тупик, в отличие от текущих auxiliary local notes/chips.

## Scope по платформам

### 1. Backend / read-model

#### `BE-SPEC-05`
Нужно зафиксировать:
- storage shape для попыток готовки;
- как вычисляется `mastery_status`;
- как это попадает в list/detail read-model;
- как не спутать `card_status` и `mastery_status`;
- какие поля нужны для будущих metrics queries.

#### `BE-IMPL-05`
Нужно реализовать:

- mutation для создания новой попытки;
- detail/list read-model поля mastery loop;
- совместимость с текущим clarification flow через `parent_recipe_id`.

### 2. iOS

#### `IOS-10`
Добавить на detail screen mastery block:
- `готовил N раз`;
- `mastery_status`;
- последняя заметка после готовки;
- CTA `Отметить готовку`.

#### `IOS-11`
Добавить лёгкий post-cook flow:
- выбор результата попытки;
- короткая заметка;
- сохранение без тяжёлого редактора.

#### `IOS-12`
Переиспользовать текущий CTA `Попросить уточнить` как contextual clarification после неудачной/неполной попытки.

### 3. Android
- Сделать parity после соответствующей backend-реализации.
- Scope parity:
  - mastery block на detail;
  - attempt logging;
  - post-cook note;
  - contextual clarification.

### 4. Metrics
- Не обязательно делать полноценную аналитику в этом пакете, если это тормозит delivery.
- Но storage/read-model должны позволить потом посчитать:
  - долю рецептов с хотя бы одной попыткой;
  - долю рецептов со второй попыткой;
  - долю `замастерил`;
  - долю рецептов с post-cook note;
  - долю clarification requests после attempt.

## Важные ограничения
- Не смешивать `mastery_status` и `card_status`.
- Не перегружать detail screen вторым тяжёлым редактором.
- Не заменять существующую `личную заметку` post-cook note-механикой.
- Не делать в этом пакете personal stable version (`REPEAT-05`).
- Не делать structured cooking mode.
- Не делать reminders, social mechanics или глубокую аналитику.
- Не превращать mastery loop в сложный журнал всех прошлых сессий на уровне UI.

## Acceptance criteria
- У полученного рецепта есть отдельный `mastery_status`.
- Пользователь может отметить факт первой и повторной готовки.
- После готовки можно сохранить короткую заметку.
- На detail screen видно:
  - количество попыток;
  - текущий mastery progress;
  - последнюю post-cook note.
- После неудачной или неполной попытки есть мягкий CTA на уточнение у автора.
- Loop не ломает текущую модель `оригинал рецепта + личная заметка`.
- UX остаётся лёгким и не превращается в тяжёлый cooking journal.

## Не делаем в этом пакете
- Полноценный structured editor.
- Merge оригинала и личной версии.
- Пошаговый cooking mode.
- Отдельный журнал всех попыток с timeline UI.
- Напоминания.
- Социальные механики.
- Deep analytics dashboard.

## Важные продуктовые ориентиры для Claude
- Пользователь должен чувствовать не "я веду журнал", а "я уже ближе к тому, чтобы это стабильно получалось".
- Первая попытка не должна ощущаться как провал; продукт должен мягко поддерживать следующую репетицию.
- Уточнение у автора — это не новая большая коммуникация, а один конкретный человеческий вопрос.
- `Mastery loop` должен усиливать смысл FamilyRecipes как продукта про близость, повторяемость и навык, а не уводить в generic cooking tracker.
