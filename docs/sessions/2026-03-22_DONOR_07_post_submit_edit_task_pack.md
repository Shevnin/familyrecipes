# Сессия: DONOR-07 Post-Submit Edit Task Pack

Дата: 2026-03-22

## Цель
Реализовать для донора ограниченный post-submit flow редактирования уже отправленного рецепта: после успешного submit на `web-reply` пользователь должен иметь возможность нажать `Исправить рецепт`, открыть prefilled форму по отдельному `edit_token`, внести правки и сохранить их без логина и без установки приложения.

## Продуктовое решение
- Это **не** постоянное редактирование по исходной request-link.
- Это отдельный reopen flow по `edit_token`, доступный ограниченное время.
- Стартовое окно редактирования: **72 часа** после первого submit.
- Редактируются только donor-originated поля:
  - `submitted_by_name`
  - `recipe_title`
  - `original_text`
  - `recipe_story`

## Scope

### `DONOR-07A` Backend contract for donor edit
- Спроектировать и реализовать хранение `edit_token` и времени истечения окна редактирования.
- После первого успешного submit создавать/сохранять `edit_token` для ответа.
- Добавить public endpoint для чтения edit-meta по `edit_token`.
- Добавить public endpoint для обновления уже отправленного donor response по `edit_token`.

### `DONOR-07B` Web-reply success state
- Обновить success-state на `web-reply/index.html`.
- Показать короткое сообщение:
  - рецепт отправлен;
  - в течение 72 часов можно внести правки;
  - CTA `Исправить рецепт`.
- CTA должен открывать edit-flow без ручного копирования ссылок.

### `DONOR-07C` Edit form UX
- Добавить отдельный режим страницы для `edit_token`.
- В edit mode форма должна:
  - загружать текущие значения;
  - быть prefilled;
  - визуально объяснять, что это исправление уже отправленного рецепта;
  - сохранять изменения без повторного создания recipe record.
- После успешного update показывать отдельный success-state: `Изменения сохранены`.

### `DONOR-07D` State coverage
- Нужны состояния:
  - invalid edit link;
  - expired edit window;
  - edit form;
  - saving;
  - edit success;
  - generic network/server error.

## Рекомендуемый backend shape
- Не обязательно следовать буквально, если Claude найдёт более чистую реализацию, но продуктовый контракт должен сохраниться.

Возможный shape:
- В `recipes` или связанной таблице хранить:
  - `edit_token_hash`
  - `edit_expires_at`
- Новый endpoint:
  - `get-edit-meta`
- Новый endpoint:
  - `update-submitted-recipe`

Требования:
- raw token в БД не хранить;
- сравнение делать по hash, как и в request token flow;
- update должен менять существующий recipe, а не создавать новый;
- update не должен ломать связь request -> recipe;
- после истечения окна endpoint возвращает стабильную ошибку `edit_window_expired`;
- при невалидном токене возвращает `edit_not_found` или эквивалентную стабильную ошибку.

## Важные ограничения
- Не ломать текущий одноразовый submit по обычному request token.
- Не превращать `submit-request` в смешанный endpoint для create/update, если это ухудшает читаемость и поддержку.
- Не добавлять donor auth.
- Не внедрять полноценную историю версий в этом пакете.
- Не менять в этом пакете donor contract v2 (`recipe_text` + `donor_comment`) — это отдельный шаг.

## Acceptance criteria
- После первого успешного donor submit пользователь видит CTA `Исправить рецепт`.
- CTA открывает prefilled edit-flow без логина.
- В течение 72 часов донор может изменить `submitted_by_name`, `recipe_title`, `original_text`, `recipe_story`.
- После сохранения рецепт обновляется в существующей записи, без дубля.
- После истечения окна edit link больше не работает и показывает понятное состояние.
- Исходная request-link остаётся одноразовой и не становится каналом бесконечного редактирования.
- Ошибки имеют стабильные пользовательские состояния, а не сырой backend text.

## Что проверить после реализации
1. Создать запрос из приложения.
2. Открыть donor link и отправить рецепт.
3. Убедиться, что success-state показывает `Исправить рецепт`.
4. Нажать `Исправить рецепт`, проверить prefilled значения.
5. Изменить текст рецепта и `Историю рецепта`, сохранить.
6. Проверить, что в приложении после refresh видны обновлённые данные, а duplicate recipe не появился.
7. Проверить поведение после искусственного истечения окна редактирования.
