# Сессия: DONOR-02 about block на donor web-reply

Дата: 2026-03-22

## Контекст
- На живом тесте donor flow подтвердилось, что тема `история рецепта` воспринимается как тёплая и ценная часть продукта.
- Пользователь хочет добавить на donor web page (`web-reply/index.html`) отдельный текстовый блок, который **объясняет смысл приложения**, а не только просит заполнить форму.
- Этот блок нужен для **первого визита донора по ссылке из мессенджера**: человек должен понять, что это не безликая форма, а способ сохранить рецепт близкого человека вместе с контекстом.

## Что нужно сделать
Добавить на submit-страницу донора **нижний explanatory block** с коротким тёплым текстом про идею FamilyRecipes.

## Где именно
- Файл: `web-reply/index.html`
- Состояние: только `state-form` в **обычном submit mode**
- Placement: **внизу страницы**, под основной карточкой формы
- Блок не должен появляться:
  - в `edit mode` (`?edit_token=...`)
  - на success screen
  - на error states

## Зачем этот блок
- Пояснить донору, что это за продукт и почему тут есть поле `История рецепта`
- Поддержать “ламповую” часть ценности: не просто инструкция, а память о человеке и контексте
- Одновременно сохранить low-friction ощущение: без установки, без сложной анкеты, можно написать своими словами

## Готовый текст

### Заголовок
`Зачем это`

### Основной текст
`FamilyRecipes помогает сохранять рецепты близких людей — семьи, друзей и просто тех, с кем хочется обмениваться своими рецептами — вместе с их историей: откуда рецепт взялся, кто его готовил и почему он стал особенным.

Вам не нужно устанавливать приложение или заполнять сложную форму — достаточно написать рецепт своими словами, как вы обычно рассказали бы его в сообщении.

Если захотите, добавьте и историю рецепта. Так у того, кто попросил рецепт, останется не только инструкция, но и память о человеке и моменте.`

## UI-направление
- Блок должен ощущаться спокойным, тёплым и вторичным по важности относительно формы
- Не превращать его в яркий маркетинговый баннер
- Визуально отделить от формы через мягкую подложку / рамку / отступ
- Текст сделать легко читаемым на мобильном экране
- Заголовок короткий, body в 2-3 коротких абзаца

## Рекомендуемая реализация
- Добавить под `.card` в `#state-form` отдельный блок, например:
  - контейнер `about-block`
  - заголовок `about-title`
  - текст `about-text`
- В edit mode скрывать этот блок через `display: none` или отдельный класс/атрибут
- Не трогать существующую submit/edit логику, API-контракты и success/error state machine

## Важные ограничения
- Не двигать форму выше по сложности
- Не вставлять этот текст между полями и CTA
- Не заменять им верхний form-context (`вас просят поделиться рецептом`)
- Не менять copy success-state в этом пакете
- Не добавлять новую backend-логику

## Acceptance criteria
- В submit mode под формой появился отдельный explanatory block
- Блок объясняет концепцию FamilyRecipes человеческим языком
- В тексте есть:
  - семейный контекст
  - простота для донора
  - смысл поля `История рецепта`
- В edit mode блок не показывается
- Success и error states не изменились
- Верстка остаётся мобильной и не ломает текущий flow
