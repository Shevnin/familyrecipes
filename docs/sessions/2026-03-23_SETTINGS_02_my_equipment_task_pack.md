# Сессия: SETTINGS-02 — `Моё оборудование` в iOS Settings

Дата: 2026-03-23

## Контекст
- Во вкладке `Ещё` в iOS-приложении уже есть отдельные settings-entrypoints, которые открывают самостоятельные экраны.
- Похожий паттерн уже реализован для `Семья и друзья`:
  - список;
  - add / edit / delete;
  - local persistence через `UserDefaults`;
  - отдельный store и отдельный edit screen.
- В `docs/product_brief.md` уже зафиксировано, что хороший рецепт должен учитывать домашнюю технику пользователя:
  - плита;
  - духовка;
  - блендер;
  - гриль;
  - и т.д.
- Сейчас в продукте нет места, где пользователь может явно указать, какое оборудование у него есть дома.

## Задача
Добавить в iOS-приложение новый пункт `Моё оборудование` во вкладке `Ещё`, чтобы пользователь мог завести и поддерживать свой список кухонной техники / оборудования.

Это supporting settings feature, а не backend или recipe-generation пакет.

## Цель
Создать аккуратную локальную настройку, где пользователь может:
- посмотреть список своей техники;
- добавить новую позицию;
- отредактировать существующую;
- удалить ненужную;
- сохранить данные между перезапусками приложения.

## Где это должно жить
- Путь входа: вкладка `Ещё` -> новый пункт `Моё оборудование`
- Вероятные файлы:
  - `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/App/MainTabView.swift`
  - новый store, например:
    - `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Core/Persistence/EquipmentStore.swift`
  - новый экран списка, например:
    - `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Features/Settings/EquipmentView.swift`
  - новый экран создания / редактирования, например:
    - `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Features/Settings/EquipmentEditView.swift`

В качестве референса по архитектуре и UX использовать существующие:
- `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Features/Settings/ContactsView.swift`
- `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Features/Settings/ContactEditView.swift`
- `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Core/Persistence/ContactsStore.swift`
- `apps/ios/FamilyRecipes/FamilyRecipes/FamilyRecipes/Core/Persistence/LocalStore.swift`

## Рекомендуемая MVP-модель данных
Не нужно делать тяжёлую структуру.

Минимально одна единица оборудования должна содержать:
- `id`
- `name` — обязательное поле
- `note` — опциональное поле

Примеры `name`:
- `Духовка`
- `Погружной блендер`
- `Аэрогриль`
- `Мультиварка`
- `Гриль-сковорода`

Примеры `note`:
- `без конвекции`
- `чаша 5 л`
- `маленькая духовка`
- `есть режим гриль`

Если Claude увидит сильную причину добавить ещё одно простое поле вроде `type`, это допустимо, но MVP лучше держать лёгким и не усложнять форму без явной пользы.

## Что нужно сделать

### 1. Добавить новый navigation item в Settings
- Название пункта: `Моё оборудование`
- Иконка: спокойная, бытовая, не слишком техничная
- Пункт должен жить в том же визуальном блоке, что остальные settings items
- Не ломать текущую структуру `Ещё`

### 2. Создать локальный store для оборудования
- По тому же паттерну, что `ContactsStore`
- Local-only persistence через `UserDefaults` / `LocalStore`
- Без backend sync
- Без Supabase
- Без cross-device merge

### 3. Создать экран списка оборудования
- Если список пуст:
  - показать аккуратный empty state
  - объяснить, что сюда можно добавить технику, которая есть дома
- Если список не пуст:
  - показать список элементов
  - поддержать переход в edit screen по тапу
  - поддержать удаление через swipe-to-delete
- На navigation bar:
  - `+` для добавления новой позиции

### 4. Создать экран создания / редактирования
- Минимальный вариант:
  - поле `Название`
  - поле `Заметка` (опционально)
- `Название` обязательно
- Нужны оба режима:
  - создать новую позицию
  - отредактировать существующую
- Сохранение должно переиспользовать один и тот же store

## UX-направление
- Держать экран простым и “домашним”, без ощущения inventory-management системы
- Не делать перегруженный catalog UI
- Не добавлять сложные иконки по типам техники, если это начинает раздувать scope
- Визуально следовать текущему warm paper language iOS-приложения
- Лучше сделать clean CRUD-поток, чем полу-структурированный “умный” редактор

## Предлагаемый empty-state copy
Title:
`Пока пусто`

Message:
`Добавьте технику и кухонные приборы, которые есть у вас дома. Позже это поможет точнее подстраивать рецепты под вашу кухню.`

Если в процессе реализации потребуется смягчить формулировку про будущее использование, это допустимо.

## Важные ограничения
- Не трогать backend
- Не подключать это к request flow в этом пакете
- Не подключать это к recipe detail в этом пакете
- Не делать suggestions / chips / автокомплит по технике в других экранах
- Не делать Android parity в той же задаче
- Не делать сложную taxonomy оборудования
- Не добавлять brand/model/capacity как обязательные структурированные поля

## Acceptance criteria
- Во вкладке `Ещё` появился новый пункт `Моё оборудование`
- По тапу открывается отдельный экран со списком оборудования
- Пользователь может:
  - добавить позицию;
  - открыть её на редактирование;
  - изменить;
  - удалить
- Список сохраняется локально между перезапусками приложения
- Empty state и edit flow не выбиваются из текущего визуального языка
- Реализация по структуре сопоставима с `Семья и друзья`, а не с отдельной новой микро-архитектурой

## Что не делаем в этом пакете
- Backend storage для оборудования
- Sync между устройствами
- Использование оборудования в recipe generation / adaptation
- Smart suggestions по типам техники
- Import готового каталога устройств
- Android implementation
