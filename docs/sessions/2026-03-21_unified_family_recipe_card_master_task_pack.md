# Сессия: Unified family recipe card (master task pack)

Дата: 2026-03-21

## Контекст
- После работы с прототипом уточнена базовая продуктовая модель.
- Раньше приложение разделяло:
  - `запросы на рецепты`;
  - `рецепты`.
- Теперь принято решение: по смыслу это **одна сущность**, одна карточка семейного рецепта, которая просто находится в разных состояниях.
- Уже зафиксированные решения:
  - запросы и рецепты должны жить в **одном списке**;
  - в primary UI **скрываем даты**;
  - нужна возможность **удаления карточки**;
  - в request flow добавляется поле **`история рецепта`**;
  - в detail screen `история рецепта` показывается **отдельным блоком над основным текстом**;
  - далее, отдельным следующим шагом, остаётся donor reply v2: `recipe_text + donor_comment`.

## Master Task Pack для Claude

### Цель
Перестроить FamilyRecipes вокруг **единой карточки семейного рецепта** вместо раздельного восприятия "запросов" и "рецептов", и подготовить backend + iOS + Android + web-reply под эту модель.

Итог должен быть таким:
- пользователь видит **один список**;
- в этом списке есть и pending-запросы, и уже полученные рецепты;
- они отличаются статусом и действиями, а не отдельными разделами;
- дата не доминирует в интерфейсе;
- у карточки есть поле `история рецепта`;
- карточку можно удалить.

### Главная продуктовая модель

#### Одна сущность
Не считать `request` и `recipe` двумя разными пользовательскими объектами.

Для пользователя есть одна сущность:
- **карточка семейного рецепта**

Она может находиться в состояниях:
- `запрошен`
- `получен`
- `уточняется`

Внутренние backend-сущности могут оставаться раздельными, если это удобно реализации, но клиентская read-model должна быть единой.

#### Что должно быть видно в UI
- название блюда;
- автор / у кого просили;
- статус;
- `история рецепта`, если есть;
- основной текст рецепта, если уже получен;
- личная заметка пользователя;
- действия по контексту состояния.

#### Что должно уйти из primary UI
- акцент на `created_at`;
- отдельный смысловой разрыв между "историей запросов" и "списком рецептов".

## Scope

### 1. Backend / read-model

#### `BE-SPEC-04`
Нужно зафиксировать backend-спеку для unified family recipe card:
- единая read-model для списка;
- единая read-model для detail;
- статус карточки;
- `recipe_story`;
- delete contract.

Нужно определить:
- из каких таблиц/полей собирается unified card;
- как вычисляется статус:
  - pending request;
  - received recipe;
  - clarification request in progress;
- как связать:
  - request;
  - original recipe;
  - follow-up clarification request;
- что считается удалением:
  - soft delete или hide from household view;
  - что происходит со связанными request/recipe rows;
  - что делать с already submitted donor data.

#### `BE-IMPL-04`
После спеки реализовать:
- unified list query / read-model;
- unified detail query / read-model;
- хранение `recipe_story`;
- delete endpoint / mutation;
- обновление API-контрактов, если нужно.

Важно:
- не ломать текущие auth/RLS гарантии;
- не ломать существующий public link flow;
- не разрушить возможность будущего donor reply v2.

### 2. iOS

#### `LIST-01`
Перестроить iOS UI так, чтобы:
- pending-запросы и готовые рецепты были в одном списке;
- не было отдельного admin-like блока history как отдельного смыслового мира;
- карточки отличались статусом, а не местом на экране.

#### `LIST-02`
Убрать даты из primary metadata:
- в list;
- в detail.

Если дата технически ещё доступна, её можно оставить только во второстепенном контексте или не показывать вовсе.

#### `REQ-09`
В create-request flow добавить поле:
- `История рецепта`

Смысл поля:
- семейный контекст;
- происхождение блюда;
- почему этот рецепт важен;
- что важно не потерять в передаче.

Это не служебное поле и не техническая заметка.

#### `DETAIL-10`
В detail screen показать `история рецепта` отдельным блоком:
- выше основного текста рецепта;
- визуально как смысловой вводный контекст;
- близко по духу к provided reference image;
- без превращения в тяжёлую декоративную карточку.

#### `CARD-DELETE-01`
Добавить удаление карточки:
- явное destructive action;
- confirmation dialog;
- корректное обновление списка после удаления.

### 3. Android
- Реализовать ту же unified card model после завершения `ANDROID-01...ANDROID-08`.
- Android не должен повторять старое раздельное мышление "запросы отдельно / рецепты отдельно".
- Scope parity:
  - единый список;
  - скрытые даты;
  - `история рецепта` в request/detail;
  - удаление карточки.

### 4. Web-reply
На этом пакете не нужно радикально переделывать donor web flow.

Нужно только учесть совместимость:
- если `recipe_story` участвует в request context, donor page может использовать его как read-only context block в будущем;
- не нужно сейчас заставлять донора редактировать `история рецепта`, если это не требуется контрактом.

Если для нового contract удобно показать donor'у этот контекст на landing/form screen, это допустимо, но не должно усложнить submit flow.

## Ограничения
- Не ломать текущий рабочий backend flow create-request -> web link -> submit -> app refresh.
- Не вводить тяжёлую многоуровневую доменную модель, если read-model можно собрать проще.
- Не превращать unified list в сложную CRM-панель со множеством бейджей и таблиц.
- Не возвращать даты как основной UI-якорь.
- Не смешивать `история рецепта` и `donor_comment`.
- Не путать:
  - `recipe_story` = контекст, заданный со стороны повара / карточки;
  - `donor_comment` = личная ремарка донора в будущем v2 контракте.

## Acceptance criteria
- В приложении есть **один список** карточек семейных рецептов.
- Pending-запросы и полученные рецепты отображаются вместе.
- Карточки различаются статусами и действиями, а не отдельными экранами/разделами.
- Даты не являются primary metadata в list/detail.
- В request flow есть поле `История рецепта`.
- В detail screen `История рецепта` показана отдельным блоком над основным текстом рецепта.
- Карточку можно удалить через подтверждение.
- Удаление не ломает household данные и не создаёт очевидных backend inconsistencies.
- Текущий flow share link -> donor submit -> recipe in app по-прежнему работает.
- Документация после выполнения отражает финальное решение, а не промежуточную версию.

## Не делаем в этом пакете
- Полную реализацию donor reply v2 (`recipe_text + donor_comment`) если она мешает завершить unified card model.
- Мультирецепты.
- Reopen/edit-token flow.
- Deep-link правки по блокам.
- Полноценную миграцию contacts/chips в backend, если она не нужна для этого конкретного изменения.
- Новый глобальный redesign продукта с нуля.

## Важные продуктовые ориентиры для Claude
- Пользователь должен ощущать, что он работает не с журналом технических событий, а с семейными блюдами.
- Pending-запрос — это не "админ-лог", а "ещё не заполненная карточка рецепта".
- `История рецепта` — это эмоционально и смыслово важная часть продукта.
- Интерфейс должен стать ближе к модели:
  - "у меня есть блюда моей семьи"
  - а не к модели:
  - "у меня отдельно есть формы, запросы, ответы и записи".
