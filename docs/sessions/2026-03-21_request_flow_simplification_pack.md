# Сессия: Request flow simplification pack

Дата: 2026-03-21

## Контекст
- После сборки живого iOS-приложения уточнён целевой UX для вкладки `Запросить рецепт`.
- Текущая реализация оказалась слишком тяжёлой для главного сценария.
- Поле `История рецепта` в запросе больше не нужно в primary flow.
- Главная задача экрана: быстро получить ссылку и отправить её человеку.

## Что хочет пользователь
Нужно упростить вкладку `Запросить рецепт` до короткого сценария:

1. Поле `Кому отправить?`
   - должно быть **редактируемым выпадающим списком**;
   - контакты редактируются:
     - прямо в самом списке;
     - и во вкладке друзей в разделе `Ещё`.

2. Поле `Какой рецепт?`
   - переименовать в `Название рецепта`.

3. Поле `История рецепта`
   - убрать из primary request flow.

4. Главная кнопка
   - не `Отправить запрос`, а `Получить ссылку`.

5. После нажатия на `Получить ссылку`
   - не уводить пользователя в отдельный success screen;
   - не перестраивать экран в отдельное состояние "успех";
   - на том же экране под кнопкой показать сформированную ссылку;
   - справа от ссылки показать действие `Копировать`;
   - под ссылкой показать пояснение:

`Вставьте ссылку в мессенджер и отправьте адресату. У него будет форма чтобы заполнить. Ваш список рецептов пополнился этим запросом.`

6. Ниже должна быть кнопка:
   - `Сформировать новый запрос`

## Master Task для Claude

### Цель
Переделать request flow в iOS под более короткий и практичный сценарий получения ссылки без отдельного success-state.

### Scope

#### `REQ-10`
Убрать поле `История рецепта` из primary request UI.

#### `REQ-11`
Сделать `Кому отправить?` редактируемым выпадающим списком.

#### `REQ-12`
Синхронизировать редактирование контактов между:
- выпадающим списком на request screen;
- разделом друзей во вкладке `Ещё`.

#### `REQ-13`
Переименовать поле:
- `Какой рецепт?` -> `Название рецепта`

#### `REQ-14`
Переименовать primary CTA:
- `Отправить запрос` -> `Получить ссылку`

#### `REQ-15`
Убрать отдельный success screen.

#### `REQ-16`
После получения ссылки показать её inline на том же экране.

#### `REQ-17`
Показать справа от ссылки действие:
- `Копировать`

#### `REQ-18`
Под ссылкой показать пояснение и кнопку:
- `Сформировать новый запрос`

### Важные ограничения
- Не ломать backend create-request flow.
- Не ломать существующую генерацию share-link.
- Не превращать request screen снова в длинную форму.
- Не возвращать тяжелый success card с множеством кнопок.
- Экран должен работать как быстрый utility flow.

### Что можно менять
- `CreateRequestView`
- `CreateRequestViewModel`
- UI-модель локального состояния экрана
- отображение контактов и интеграцию с existing contacts storage
- inline result composition

### Что менять осторожно
- backend contract `create-request`
- shared contacts persistence
- create-request side effects, которые уже влияют на unified list

### Acceptance criteria
- На экране больше нет поля `История рецепта`.
- Поле `Кому отправить?` работает как редактируемый dropdown.
- Поле называется `Название рецепта`.
- Кнопка называется `Получить ссылку`.
- После нажатия пользователь остаётся на том же экране.
- Ссылка показывается inline под кнопкой.
- Рядом со ссылкой есть `Копировать`.
- Ниже есть поясняющий текст.
- Ниже есть кнопка `Сформировать новый запрос`.
- Новый запрос по-прежнему создаёт карточку в unified family recipe list.

### Не делаем
- Donor reply v2.
- Новый backend refactor.
- Новый contact sync backend.
- Android реализацию в этом же пакете, если это мешает быстро довести iOS UX.
