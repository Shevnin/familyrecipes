# Сессия: Donor post-submit edit flow

Дата: 2026-03-22

## Контекст
- После живого теста donor flow получен качественный сигнал: заполнение формы удобно, `история рецепта` воспринимается как ценная часть ответа.
- Одновременно появился конкретный продуктовый запрос: дать донору возможность исправить ответ после отправки, если он что-то забыл.
- Пользователь принял решение идти не в сторону постоянного редактирования, а в сторону узкого и понятного **post-submit flow**.

## Принятое решение
- После успешной отправки donor success-state должен показывать действие `Исправить рецепт`.
- Кнопка ведёт не обратно по исходной request-link, а в отдельный prefilled web-flow по `edit_token`.
- Редактирование ограничено по времени. Стартовое продуктовое окно: **72 часа**.
- Flow остаётся **zero-install** и **без логина/регистрации**.
- Редактируются только donor-originated поля текущего ответа:
  - `submitted_by_name`
  - `recipe_title`
  - `original_text`
  - `recipe_story`
  - позже также `donor_comment`, когда появится v2 contract

## Что сознательно не делаем
- Не превращаем это в полноценный versioned editor.
- Не разрешаем бесконечное редактирование по исходной request-link.
- Не добавляем аккаунт, email verification или auth для донора.
- Не решаем в этом пакете мультирецепты и не связываем это с `DONOR-06`.

## Рекомендуемый UX
1. Донор отправляет рецепт.
2. Видит success-state:
   - `Рецепт отправлен`
   - короткое пояснение, что в течение 72 часов можно внести правки;
   - CTA `Исправить рецепт`
3. По CTA открывается форма с уже заполненными значениями.
4. После сохранения показывается состояние `Изменения сохранены`.
5. После истечения окна редактирования reopen больше недоступен.

## Почему это лучший MVP-путь
- Решает реальную боль "вспомнил позже / забыл дописать".
- Не увеличивает барьер первого submit.
- Не ломает основной mental model: сначала отправил, потом при необходимости поправил.
- Не требует тяжёлой модели совместного редактирования.
