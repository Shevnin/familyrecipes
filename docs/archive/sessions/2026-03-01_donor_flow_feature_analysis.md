# Сессия: Donor request flow — feature analysis

Дата: 2026-03-01

## Входной список идей
1. Улучшенный `share_text` (объяснить, что без установки/регистрации, 2-3 минуты).
2. Onboarding-блок на `web-reply` (что от человека хотят, это быстро и просто).
3. Режимы ответа: `Коротко` / `Подробно`.
4. Черновик на стороне донора (не потерять текст при закрытии вкладки).
5. Reminder из приложения, если запрос не закрыт.
6. Мультирецепты по одной ссылке (`+ Добавить еще рецепт`).
7. Повторное редактирование после submit (reopen по edit-token, time window).
8. Запрос правок по блокам (ингредиенты/шаги/советы) с deep-link в нужный блок.

## Разбор: что оставить, что убрать, что улучшить

### Оставить в ближайшем цикле (высокая ценность / низкая сложность)
- `DONOR-01` Share text v2.
- `DONOR-02` Onboarding блок на web-reply.
- `DONOR-03` Режимы `Коротко/Подробно`.
- `DONOR-04` Autosave черновика (localStorage + restore banner).

Причина:
- Эти фичи напрямую поднимают конверсию submit и не требуют тяжелых backend-изменений.

### Оставить, но после Android parity (средняя сложность)
- `DONOR-05` Reminder на pending-запрос.
- `DONOR-06` Мультирецепты по одной ссылке, но в формате "серия ответов в одном request context", не "одна запись request -> много блюд без структуры".
- `DONOR-07` Reopen/редактирование после submit (ограниченное окно, например 72 часа).

Причина:
- Нужны изменения контракта и модели данных; важно сделать после закрытия Android `ANDROID-01...08`.

### Отложить (или убрать из MVP)
- `DONOR-08` Правки по блокам (deep-link сразу в ингредиенты/шаги/советы).

Причина:
- Сильная UX/BE сложность: версия рецепта, merge конфликтов, стабильные блок-id. Для текущего этапа избыточно.

## Улучшения к исходным идеям

### Улучшение для мультирецептов
- Не делать "бесконечную форму в одном submit".
- Делать последовательные submit в рамках одной "сессии донора":
  - экран успеха -> CTA `Добавить еще рецепт`;
  - каждое блюдо сохраняется отдельным recipe record;
  - связь через `request_id` + `batch_id`.

### Улучшение для reopen
- Ограничить по времени + по числу reopen (например 3 правки за окно).
- Показывать пользователю последний timestamp обновления.
- На стороне повара отображать "обновлено донором".

### Улучшение для reminder
- Добавить шаблоны tone: мягкий/нейтральный.
- Защититься от спама: cooldown (например 12 часов).

## Что дополнить (новые идеи, которых не хватало)
1. `DONOR-09` Индикатор прогресса заполнения (например "1 из 2 шагов").
2. `DONOR-10` "Пример хорошего ответа" в collapsed-help (для людей, которые не понимают, как писать рецепт).
3. `DONOR-11` Метрики воронки донора:
   - open_link
   - form_started
   - submit_success
   - submit_error
   - retry_submit

## Рекомендуемый порядок реализации
1. `DONOR-01` Share text v2.
2. `DONOR-02` Onboarding block.
3. `DONOR-03` Режимы коротко/подробно.
4. `DONOR-04` Autosave draft.
5. Проверка метрик конверсии (`DONOR-11`).
6. Затем backend-усложнения: `DONOR-05...07`.

## Риски
- Без `DONOR-11` сложно доказать эффект улучшений.
- `DONOR-06/07` без четкой модели версий могут привести к дублям и путанице в рецептах.
